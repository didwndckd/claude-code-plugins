# spec 분석 스킬

과제 요구사항/안내 문서를 분석해서 후속 작업(기술 스택 선택, 아키텍처 결정, 작업 분리 등)의 기반이 되는 base spec 문서를 작성한다.

## 산출물

- 경로: `docs/specs/base.md`
- 위 파일이 없으면 새로 만든다. 이미 있으면 사용자에게 덮어쓸지 확인한 후 진행한다.

## 작업 절차

### 1. 입력 수집

- `spec_inputs` 배열의 각 원소를 순회하며 다음 규칙으로 분류·수집한다.
  - **파일 경로로 보이는 항목** (예 `.pdf`, `.md`, `.txt`, `.png`, `./...`, 절대경로 등): Read 도구로 읽는다. PDF·이미지도 Read로 처리.
  - **본문 텍스트/메모로 보이는 항목**: 그 내용을 그대로 컨텍스트에 포함.
  - 모호하면 user에게 한 번 확인 후 분류.
- 수집한 입력은 **출처를 보존**한다. 후속 단계에서 base.md `> 출처:` 라인과 각 항목 인용에 사용.
- 입력이 여러 개일 때는 서로 충돌하는 요구가 있는지 점검:
  - 충돌 발견 시 임의 결정 금지. "user 결정 필요 항목"에 질문으로 남긴다.
  - 동일 요구가 중복되면 한 번만 기록하되 출처 모두 표기.

### 2. 참고 문서 보관

분석 전에 **파일** 입력을 레퍼런스 폴더(`docs/specs/references/`)로 이동시켜 둔다. 텍스트로 직접 전달된 입력은 옮길 대상이 없으므로 이 단계 스킵.

- `docs/specs/references/` 폴더가 없으면 먼저 생성 (상위 `docs/specs/`도 함께).
- 이동은 `mv`를 사용 (사본이 아니라 원본 자료를 레퍼런스 폴더로 모으는 목적).
- 동일 파일명이 이미 존재하면 덮어쓰지 말고 user에게 어떻게 처리할지 확인 (rename / overwrite / skip).
- 권한·심볼릭 링크·다른 디스크 등으로 이동이 실패하면 멈추고 user에게 보고. 임의로 cp 후 원본 삭제로 우회하지 않는다.
- 이동 후에는 **새 경로를 출처로 사용**한다. base.md `> 출처:` 라인과 인용 모두 `docs/specs/references/<파일명>` 기준. 원본 위치는 별도 메타로 한 줄 부기 (`> 원본 위치: ...`).

### 3. 분석 항목 정리

다음 4가지 섹션을 빠짐없이 채운다. 문서에 명시되지 않은 항목은 "명시되지 않음"으로 남기고 추측으로 메우지 않는다.

1. **요구 기술 스택**
   - 언어/플랫폼/최소 버전, UI 프레임워크, 반응형/비동기 라이브러리, 네트워크 라이브러리, 기타 라이브러리 제약.
2. **요구 아키텍처 / 디자인 패턴**
   - Clean Architecture, MVVM, MVC 등 명시된 패턴.
   - 명시되지 않은 경우 "자유"로 기록.
3. **기능 목록**
   - 화면/플로우 단위로 분리. 각 항목은 한 줄 요약 + 핵심 동작 bullet.
   - 우선순위/필수·선택 여부가 문서에 있으면 함께 표기.
4. **추가 결정해야 할 사항**
   - 현재 기능에 추가로 고려할 부분(엣지 케이스, UX 결정 포인트, 비기능 요구사항).
   - 예상되는 이슈/리스크(API 제한, 인증, 페이지네이션, 캐시, 접근성 등).
   - 후속 단계에서 user가 결정해야 할 질문 목록.

### 4. 문서 작성

`docs/specs/base.md`에 아래 템플릿으로 작성한다.

```markdown
# 과제 base spec

> 출처: <옮긴 후 경로 또는 텍스트 인용. 예: `docs/specs/references/assignment.pdf`, `사용자 직접 전달 메모`>
> 원본 위치: <파일 입력의 이동 전 경로. 텍스트 입력만 있으면 생략>
> 작성일: YYYY-MM-DD

## 1. 요구 기술 스택
- ...

## 2. 요구 아키텍처 / 디자인 패턴
- ...

## 3. 기능 목록
### 3.1 <기능명>
- 요약: ...
- 동작:
  - ...
- 우선순위: 필수 | 선택 | 명시되지 않음

## 4. 추가 고려사항 / 예상 이슈
### 추가 고려사항
- ...

### 예상 이슈
- ...

### user 결정 필요 항목
- [ ] ...
```

### 5. 검수 (필수)

문서 작성 직후 반드시 검수 subagent를 실행한다. 검수 없이 종료하지 않는다.

- 사용 도구: `Agent` (subagent_type=`general-purpose`)
- 검수 입력: 방금 작성한 `docs/specs/base.md` 경로와 원본 입력 출처.
- 검수 프롬프트 가이드:
  ```
  방금 작성된 spec 분석 문서를 검수해주세요.

  - 분석 문서: <ABS_PATH>/docs/specs/base.md
  - 원본 입력 목록: <spec_inputs 각 항목의 경로 또는 요약을 bullet로 나열>

  점검 항목:
  1. 4개 섹션(기술 스택 / 아키텍처 / 기능 목록 / 추가 고려사항)이 모두 채워져 있는가
  2. 원본에 명시된 요구사항이 누락되지 않았는가
  3. 원본에 없는 내용이 추측으로 들어가 있지 않은가 (있다면 어느 줄)
  4. 기능 목록이 화면/플로우 단위로 분리되어 있는가
  5. user가 결정해야 할 항목이 체크리스트로 정리되어 있는가
  6. 여러 입력 사이의 충돌이 임의로 결정되지 않고 결정 필요 항목으로 남았는가
  7. 파일로 전달된 참고 문서가 모두 `docs/specs/references/`로 이동됐고, base.md 출처가 옮긴 후 경로를 가리키는가

  발견한 누락/오류/추측 항목을 bullet로 정리해서 보고. 200자 이내 권장.
  ```
- 검수 결과를 user에게 그대로 전달하고, 수정이 필요하면 반영한 뒤 다시 한번 검수를 돌릴지 user에게 확인.

## 제약

- 추측 금지: 문서에 없는 사항은 "명시되지 않음"으로 남기거나 "user 결정 필요 항목"에 질문으로 남긴다.
- 코드 구현/폴더 구조 결정은 이 스킬의 범위가 아니다 (후속 단계 책임).
